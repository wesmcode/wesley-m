02 — The real goal

It's not "four projects."
It's one project, four moments.

wesley-m.com is a credible professional surface for Wesley's writing, resume, and design taste. Static portfolio at the root. A new Payload CMS at apps/blog/. Two legacy long-form posts to bring forward. Two competing index designs to choose between.

The agent teams aren't four parallel workstreams. They're four phases of one outcome: ship the new blog, retire the old one, keep the site fast and protected.

  • Stack Vanilla HTML/CSS/JS + Payload/Next.js
  • Hosting Vercel · apex via CNAME
  • Constraint No frameworks unless earned
  • Stance AI-scraping-protected (noai · noimageai)

08 — Collaboration model

Agents don't talk to each other.
They pass files.

The slash command turns the main session into a Supervisor. The Supervisor spawns one subagent at a time (or several in parallel), holds the shared state, and decides what runs next. Subagents never spawn other subagents.

Every handoff is a file path, not a pasted blob. The Supervisor's context stays small. The next agent reads only what it needs.

planner   →  plans/foo.plan.md
              ↓
researcher → plans/foo.sources.json
              ↓
drafter   →  drafts/foo.md
              ↓
critic    →  reviews/foo.review.json
              ↓ (ship)
formatter →  build/foo.lexical.json
              ↓
publisher →  postId, adminUrl

09 — The glue across phases

Some files are shared. Change one, and two teams adapt.

apps/blog/src/editor/
buildEditor.ts

Schema of record for custom blocks

Read by: migrate-mapper · blog-formatter

blog/*.html
voice anchors

Authoritative voice samples

Read by: blog-planner · blog-critic

audits/*.json
P0–P3 issue lists

Shared audit state

Read by: site-fixer · site-reviewer

If P1's verdict reshapes a card layout, P2's mapper and P3's formatter both inherit it through buildEditor.ts. No drift.

The PM playbook

  1. Sequence P1 → P2 → P3. Each unlocks the next.
  2. Run P4 from day one, read-only. Approve fixes manually.
  3. Every gate is a human decision, not an agent verdict.
  4. Retire teams when their work is done. Don't keep them running.

How the teams catch each other's misses

  • P1 → P2 Winning card layout feeds the migration mapper. No rework.
  • P2 → P3 Validator catches what P1 changed in the index.
  • P3 ⇄ legacy Critic uses the old posts as voice anchors.
  • P4 → all Audits everything P2 and P3 ship into the site.

14 — Recommendation

Start with /ab-evaluate.

It's the team whose output most reshapes the project. It doesn't depend on the not-yet-pushed Payload app. And its verdict tells you whether the Critic / Synthesizer pattern is calibrated before you trust it elsewhere.
